# XML Boundaries

适用于以下任务：

- 判断当前需求是否应该写 XML
- 判断是否应该回退原生 SQL
- 审查现有 XML 是否属于合理使用
- 约束 Agent 不要默认把 xbatis 项目写回传统 MyBatis 风格

## 核心原则

把 XML / 原生 SQL 视为后置选项，不是默认起点。

先按下面顺序判断：

1. 能否直接用 `MybatisMapper<T>` / `BasicMapper`
2. 能否用 `QueryChain`
3. 能否用结果映射、对象转条件、对象转排序
4. 能否用 SQL 模板、动态 where、动态 query
5. 只有前面都不合适，再考虑 XML / 原生 SQL

## 明确允许写 XML 的场景

以下场景允许考虑 XML：

1. 报表 SQL 结构极重
2. 数据库方言差异明显，且原生表达更稳定
3. 需要复用一段已有成熟 XML，且用户明确要求延续
4. 框架原生表达虽然能做，但可读性明显更差
5. 已有系统围绕 XML 沉淀了明确组织方式，当前改动必须沿用

这类场景的共性是：

- 问题不是“能不能写出来”
- 而是“框架原生写法是否不再是最清晰、最稳定的表达”

## 明确不应写 XML 的场景

以下场景默认不应写 XML：

1. 简单 CRUD
2. 标准分页列表
3. 常规动态条件查询
4. 常规 exists / count
5. 常规联表查询
6. 标准 VO 返回
7. 搜索表单筛选
8. 仅因为开发者熟悉 XML

如果这些场景仍写 XML，优先判定为实现路径偏离 xbatis 风格。

## SQL 模板与 XML 的边界

先区分“局部复杂”还是“整体复杂”。

### 局部复杂

例如：

- 某个数据库函数表达复杂
- 某个条件表达式不够顺手
- 某个计算字段需要模板包装

优先：

- SQL 模板
- 函数模板
- 条件模板

不要：

- 因为一个局部表达式复杂，就把整条查询退回 XML

### 整体复杂

例如：

- 超长报表 SQL
- 多层嵌套、窗口函数、数据库专有语法密集
- 与框架 DSL 的表达重心明显不匹配

这类场景才更适合考虑 XML / 原生 SQL。

## QueryChain 与 XML 的边界

如果需求仍然是：

- 标准 where
- 标准 order by
- 标准 group by
- 标准分页
- 标准 join
- 标准 VO 返回

那么即使查询“看起来不少”，也应优先继续留在 `QueryChain`。

不要把下面这些误判成“必须 XML”：

1. 条件比较多
2. if 分支较多
3. 有联表
4. 有分页
5. 有排序
6. 返回 VO

这些恰恰属于 xbatis 原生能力的主要覆盖面。

## 单 Mapper / 多 Mapper 与 XML 的边界

判断 XML 时，必须先尊重项目当前 Mapper 模式。

### 多 Mapper 模式

通常要求：

- XML namespace 对应具体实体 Mapper
- 自定义 SQL 放在实体 Mapper 语义下

不要：

- 在多 Mapper 项目里为了写 XML，临时引入统一 `BasicMapper`

### 单 Mapper 模式

通常要求：

- XML namespace 对应统一 `BasicMapper` 子接口
- SQL id 可能带实体名前缀
- 自定义 SQL 的调用方式与 `withSqlSession` 等统一入口一致

不要：

- 在单 Mapper 项目里为了写 XML，又新建每实体一个 Mapper

## 代码审查时如何判断 XML 是否合理

审查时，先问四个问题：

1. 这个需求是否真的无法用 `QueryChain` 或模板清晰表达
2. 当前 XML 是否顺应项目现有 Mapper 模式
3. 当前 XML 是否只是因为开发习惯而产生
4. 当前 XML 是否引入了第二套难以维护的查询体系

### 合理信号

- 报表类 SQL
- 数据库专有写法多
- 复用历史 XML 且范围受控
- 原生写法明显更可读

### 风险信号

- 简单查询写 XML
- 列表页写 XML
- 动态筛选写 XML
- 只是因为熟悉 XML
- 为了规避学习 `QueryChain`

## Agent 输出建议

如果判断“不该写 XML”，输出时优先建议：

1. 改为内置 Mapper
2. 改为 `QueryChain`
3. 改为 SQL 模板
4. 保留 XML 但压缩到最小必要范围

不要只说“建议优化”，要明确指出：

- 当前为什么不适合 XML
- 应替换成哪种 xbatis 原生能力

## 最终边界

可以记成一句话：

> 如果问题仍然属于 xbatis 的主能力覆盖面，就不要先写 XML；只有当原生表达明显失去清晰度和稳定性时，才回退 XML / 原生 SQL。
